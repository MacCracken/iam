# 2026-05-19 — iam P(-1) security audit

> **Scope**: iam @ v0.5.0 (commit `a57c17b`, tag `0.5.0`). Internal
> review per CLAUDE.md *Process P(-1)* §3 ("every mihi return checked,
> every output write bounded"), plus the broader hard-constraint list.
> This file is the M5 "security audit pass" deliverable referenced by
> the v1.0 roadmap.
>
> **Scope of *threat*** is limited to iam itself. Probe parsing
> (`/proc`, `/sys`, sysfs, PCI config space) is mihi's responsibility
> and is audited in mihi's own audit log. This document treats mihi's
> return values as a trust boundary.

## TL;DR

**Pass with one open item.** All mihi returns are checked, all output
writes are bounded, no unsafe syscall families are reachable from iam,
the binary is exit-0-on-every-path (login MOTD safe), and the test
suite (90 assertions) executes the ADR-contract behaviors that the
audit relies on. One acknowledged-but-unmitigated item: TTY escape
sanitization on mihi-returned strings (already flagged in
[`SECURITY.md`](../../SECURITY.md)).

## Threat model

From [`SECURITY.md`](../../SECURITY.md):

1. **mihi return-data abuse** — adversarially-large or malformed
   buffers from a hostile `/proc` / `/sys` / sysfs file reaching iam
   via mihi.
2. **TTY escape injection** — control sequences in mihi-returned
   strings (hostname, distro PRETTY_NAME, GPU name) reaching stdout
   verbatim.

iam does not spawn processes, open sockets, read env vars, parse
config, or open files. These categories are out of scope by
construction.

## Source review — file by file

### `src/main.cyr` — driver

**mihi return handling (CLEAN)**

| Probe                 | Return        | Check                                          |
| --------------------- | ------------- | ---------------------------------------------- |
| `mihi_uname(&uts)`    | i64 result    | `is_err_result(ures)` → sets `uok=0`; degraded path uses `iam_render(.., 0)` for Kernel + Host (renders `(unknown)`). main.cyr:46, 92–96 |
| `mihi_cpu_model(...)` | cstring or 0  | Passed directly to `iam_render`; null-handled in renderer. main.cyr:50, 79 |
| `mihi_mem_total(...)` | i64 bytes     | `if (mbytes > 0)` before formatting; otherwise `mlen=0` and `iam_render_buf` emits `(unknown)`. main.cyr:55–58, 82 |
| `mihi_distro(...)`    | cstring or 0  | Passed directly to `iam_render`; null-handled in renderer. main.cyr:62, 99 |
| `mihi_uptime_secs(...)` | i64 seconds | `if (usecs >= 0)` before formatting; -1 routes to `(unknown)`. main.cyr:65–69, 102 |
| `mihi_gpu_count()`    | i64 count     | `if (ngpu > 0)` gates the GPU line entirely; suppress-on-zero is the M3 contract (a GPU-less box is not an error). main.cyr:110–114 |
| `mihi_gpu_name(0)`    | cstring or 0  | Passed directly to `iam_render`; null-handled in renderer. main.cyr:112 |

**Every renderer's return value is also checked** via the `if (n > 0)
{ pos = pos + n; }` guard at the call site (main.cyr:79–114). On
overflow (renderer returns -1) `pos` is not advanced and subsequent
renderers see less remaining capacity — the partial output already in
`out[0..pos]` is still valid prefix per the ADR.

**Buffer caps (CLEAN — all stack-allocated, all bounded)**

| Buffer           | Size      | Used by                          |
| ---------------- | --------- | -------------------------------- |
| `uts[390]`       | 390 B     | `mihi_uname` shared UTS buffer   |
| `cpubuf[8192]`   | 8 KiB     | `mihi_cpu_model` scratch         |
| `membuf[4096]`   | 4 KiB     | `mihi_mem_total` scratch         |
| `memout[24]`     | 24 B      | `iam_format_bytes` output        |
| `distbuf[1024]`  | 1 KiB     | `mihi_distro` scratch            |
| `upbuf[64]`      | 64 B      | `mihi_uptime_secs` scratch       |
| `upout[24]`      | 24 B      | `iam_format_uptime` output       |
| `out[4096]`      | 4 KiB     | Final flushed output             |

Real-world worst-case lines (~50-byte CPU model + 8-byte label + `\n`)
≈ 60 B × 7 lines ≈ 420 B. The 4 KiB final buffer has ~10× headroom for
pathological `PRETTY_NAME` / CPU model strings; on overflow the
renderer signals -1, the driver stops appending, and the partial
prefix flushes — bounded, no UB.

**Syscalls (CLEAN — read-only and minimal)**

Direct syscalls from iam source: `syscall(SYS_EXIT, r)` (main.cyr:124).
The only output-side syscall is `print(&out, pos)` (main.cyr:119),
which expands to `write(2)`. All probe-side syscalls are issued by
mihi (`uname(2)`, reads against `/proc/cpuinfo` / `/proc/meminfo` /
`/etc/os-release` / `/proc/uptime`, PCI config space for GPU). iam
itself never calls `open`, `openat`, `fork`, `execve`, `socket`,
`connect`, or any process/network family.

### `src/display.cyr` — renderers + formatters

**Bounds on `iam_put` and `iam_copy` (CLEAN)**

- `iam_put`: `if (pos >= cap) { return 0 - 1; }` — strict bound on
  single-byte write. display.cyr:38–42.
- `iam_copy`: `if (pos + len > cap) { return 0 - 1; }` — strict bound
  on multi-byte copy. display.cyr:47–55. Both `pos` and `len` are i64;
  with cap ≤ 4096 and these being stack-buffer offsets, integer
  overflow on `pos + len` is not reachable.

**Renderer overflow handling (CLEAN)**

Every `iam_put` / `iam_copy` call in `iam_render`, `iam_render_buf`,
`iam_render_kernel` checks the return for `< 0` and short-circuits.
The `tests/iam.tcyr` overflow case (`cap=10` for a 20+ byte line)
returns `-1` and the driver elides the line — exercised at
iam.tcyr:147–150.

**Null-input handling (CLEAN)**

- `iam_render`: `if (value == 0) { ... emit IAM_UNKNOWN_TEXT ... }`
  (display.cyr:76–81). Covered by ADR-contract tests at iam.tcyr:139–144.
- `iam_render_buf`: `if (vlen <= 0)` covers both formatter-failed (0)
  and explicit-error (-1) signals (display.cyr:105–109). Tests at
  iam.tcyr:157–162.
- `iam_render_kernel`: `if (name == 0 || version == 0) { return
  iam_render(out, cap, label, 0); }` (display.cyr:121–123). Tests at
  iam.tcyr:170–174.

**Formatter bounds (CLEAN)**

- `iam_uint_into` (display.cyr:155–173): cap-too-small returns 0
  without partial writes (iam.tcyr:52–53); negative input rejected
  (iam.tcyr:55–56); digit count capped at 20 (i64.max is 19 digits,
  so we never overrun the tmp array).
- `iam_format_bytes` (display.cyr:185–208): cap < 6 short-circuit,
  negative input short-circuit, every unit/value/space write bounded
  against cap. Tests at iam.tcyr:62–86.

**Reliance on `strlen` for mihi cstrings (NOTED)**

Renderers call `strlen(label)` (label is a compile-time iam string,
trusted) and `strlen(value)` (value is a mihi-returned cstring). If
mihi ever returned a non-null-terminated buffer claiming to be a
cstring, `strlen` could walk off the end of mihi's scratch buffer.
This is a mihi-side invariant; iam relies on it. Audited in mihi's
own audit log; not actionable from iam.

### `src/uptime.cyr` — uptime formatter

**Bounds (CLEAN)**

`iam_format_uptime` (uptime.cyr:20–77) guards every `store8` with
`if (pos >= cap) { return 0; }` and every digit-write via
`iam_uint_into` (which is itself bounded). cap < 4 short-circuits
before any write. Negative `secs` is not handled explicitly but is
caught upstream by the `usecs >= 0` check in main.cyr:69 before the
formatter is called.

Edge cases covered by tests (iam.tcyr:88–111): fresh-boot `0 → "<1m"`,
sub-minute, exact-minute, zero-hour-elision (`2d 1m`), cap-too-small.

### `src/test.cyr` — test entrypoint

No probe calls, no I/O beyond `assert_summary`. Out of audit scope.

## Findings

### F-001 — TTY escape sequences in mihi-returned strings (OPEN, LOW)

**Status**: open, acknowledged in `SECURITY.md`.

**Issue**: If a mihi probe ever returns a string containing ANSI
escape sequences (e.g., a maliciously-crafted `/etc/os-release`
`PRETTY_NAME`, or a hostname containing `\x1b[`), iam will write it
to stdout verbatim. ADR 0001's "TTY == pipe" no-color guarantee
applies to iam's *own* emitted bytes; it does not currently include
sanitizing mihi-returned content.

**Severity**: LOW. Requires attacker control over a system-config
file the probe consumes. Realistically requires either root or a
boot-time compromise — at which point iam is not the attacker's
primary lever. But the SECURITY.md threat model lists it
specifically, and the v1.0 contract should either honor that or
amend the doc.

**Recommendation for v1.0**: add a control-character/escape filter at
the renderer boundary (replace any byte < 0x20 except '\n' with '?',
strip '\x1b'). Costs a single pass over the value bytes — sub-µs at
realistic value lengths. Land before the M6 freeze.

### F-002 — `strlen` on mihi cstrings is trust-dependent (NOTED, INFO)

**Status**: acknowledged dependency on mihi invariant.

**Issue**: iam calls `strlen` on cstrings returned by mihi. If mihi
ever returned a non-NUL-terminated buffer claiming to be a cstring,
`strlen` would over-read mihi's scratch buffer.

**Severity**: INFO. Not actionable from iam. Mihi's own audit log
should cover the invariant.

**Recommendation**: no change in iam. Cross-link this finding from
mihi's audit when next refreshed.

## No issues found

- Buffer over/underflow — every store8/copy is bounded.
- Integer overflow — i64 stack offsets with caps ≤ 4 KiB cannot wrap.
- Probe-failure cascade — all-probes-failed degraded shape covered by
  test (iam.tcyr:208–224); exit code stays 0; output stays six lines.
- Exit-code discipline — every code path returns 0; login `set -e`
  cannot be tripped by iam.
- Environment-input attacks — iam reads no env vars (`getenv` absent
  from source).
- Filesystem attacks — iam opens no files (`open` / `openat` absent
  from source); all file reads are mihi's responsibility.
- Network attacks — iam opens no sockets; no network code present.
