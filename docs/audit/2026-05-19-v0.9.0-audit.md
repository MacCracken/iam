## Section A — F-001 mitigation review (new for v0.9.0)

### Design

**`iam_copy_value(out, cap, pos, src, len): i64`** at
`src/display.cyr:74` — the canonical value-column copy. Sanitizes
any byte `< 0x20` to `?` (0x3F) during copy; leaves all other bytes
untouched. Returns new pos on success or `-1` on overflow (same
contract as `iam_copy`).

```cyrius
fn iam_copy_value(out, cap, pos, src, len): i64 {
    if (pos + len > cap) { return 0 - 1; }
    var i = 0;
    while (i < len) {
        var b = load8(src + i);
        if (b < 32) { b = 63; }
        store8(out + pos + i, b);
        i = i + 1;
    }
    return pos + len;
}
```

The 32 / 63 magic numbers are 0x20 (the first printable ASCII) and
0x3F (`?`) respectively, kept inline to match the surrounding code
style. The cstring NUL (`\0` = 0x00) is in the `< 0x20` set so any
NUL byte that survives `strlen` (mihi invariant violation) would
render as `?` — defense in depth against F-002.

### Coverage — call-site fan-in

The audit threat from M5/M5.5 was that mihi-returned strings (CPU
model, distro PRETTY_NAME, hostname, kernel name+version, GPU name)
could contain ANSI escapes or other control bytes. Every renderer
that takes a mihi-sourced value now routes the value bytes through
`iam_copy_value`:

| Renderer call site                                              | What gets sanitized | Notes |
| --------------------------------------------------------------- | ------------------- | ----- |
| `iam_render` value path (`display.cyr:109`)                     | cstring `value`     | Covers CPU / Host / Distro / GPU lines (every cstring-typed mihi return). |
| `iam_render_buf` value path (`display.cyr:137`)                 | buffer `valbuf`     | Covers Memory + Uptime values. iam-controlled formatters (`iam_format_bytes`, `iam_format_uptime`) only emit `[0-9 a-zA-Z]` so the sanitizer is a no-op there — but uniform sanitization simplifies the audit story to "the value column is always sanitized." |
| `iam_render_kernel` name path (`display.cyr:165`)               | cstring `name`      | uts buffer cell, mihi-controlled. |
| `iam_render_kernel` version path (`display.cyr:169`)            | cstring `version`   | uts buffer cell, mihi-controlled. |

Iam-controlled bytes — labels, padding spaces, `IAM_UNKNOWN_TEXT`
("(unknown)"), and the trailing `\n` — continue to route through
plain `iam_copy` / `iam_put`. The split is explicit so a future
auditor can answer "is this byte sanitized?" by checking which
function the call site uses.

### Test coverage

15 new assertions in `tests/iam.tcyr` under a new
`test_group("display.cyr — F-001 TTY-escape sanitization")`:

| Test                                                             | Bytes-of-interest               |
| ---------------------------------------------------------------- | ------------------------------- |
| `iam_copy_value` byte-boundary unit                              | 0x1F → `?`, 0x20 → unchanged, `A` → unchanged, 0x09 (TAB) → `?` |
| `iam_render` ESC-laden CPU value                                 | `\x1b[31mRed` → `?[31mRed`      |
| `iam_render` newline-injection attempt                           | `A\nB` → `A?B` — preserves line-equals-one-fact contract against a malicious mihi probe attempting to forge an extra line |
| `iam_render_buf` ESC mid-buffer                                  | `1 \x1bGiB` → `1 ?GiB`          |
| `iam_render_kernel` ESC in name + TAB in version                 | `\x1bLinux` + `7.0\x091` → `?Linux 7.0?1` |
| `IAM_UNKNOWN_TEXT` path unchanged                                | Locks in that the `(unknown)` fallback is not affected by F-001 |

Test inputs built via explicit `store8` calls — does not rely on
Cyrius string-literal escape support beyond `\n`. Total assertion
count: 90 (M5.5 baseline) → 105 (v0.9.0).

### Bench impact

3 trials × N=500 invocations on archaemenid:

```
trial=1 per=1511us
trial=2 per=1510us
trial=3 per=1510us
```

Median **1510 µs**. v0.5.0 baseline was 1511 µs (same noise band,
1490–1545 µs per `docs/benchmarks.md`). The sanitizer adds one
comparison + one possible store-overwrite per value byte; at
~10–50 byte value lengths × 6–7 lines per invocation, the total
added work is ~150–350 byte-ops against a ~1500 µs probe budget.
**Result: invisible at this scale.** F-001 mitigation is free on
the login hot-path.

## Findings

### F-001 — TTY escape sequences in mihi-returned strings (**RESOLVED at v0.9.0**)

**Status closed**: mitigation landed in `iam_copy_value`
(display.cyr:74) and is wired into all three renderers
(display.cyr:109, 137, 165, 169). Byte-boundary tests at
`tests/iam.tcyr` lock the behavior. Real-world bench confirms zero
measurable cost.

**Resolution detail**: replaces any value-column byte `< 0x20` with
`?` (0x3F). Neutralizes the entire C0 control range — ANSI CSI
escapes (lead byte 0x1B), TAB (0x09), bell (0x07), the
line-injection class (LF 0x0A / CR 0x0D). DEL (0x7F) and the C1
control range (0x80–0x9F) are *not* sanitized — DEL is above 0x20,
and high-bit bytes are reserved for legitimate UTF-8 sequences in
distro PRETTY_NAMEs (e.g. "Pop!_OS" or "openSUSE Tumbleweed"
variants). The audit's threat model
([`../../SECURITY.md`](../../SECURITY.md)) names ANSI escape
injection specifically, and that surface is fully covered.

**SECURITY.md** can stay as-is — its "iam must sanitize before
writing to stdout" claim is now true.

### F-002 — `strlen` on mihi cstrings is trust-dependent (INFO, carries forward)

**Status unchanged**: still informational. mihi pin remains 0.7.0
across the v0.5.0 → v0.9.0 arc; the cstring NUL-termination
invariant the renderer relies on is unchanged.

**Defense-in-depth observation**: F-001's sanitizer doesn't
*resolve* F-002 (a non-NUL-terminated mihi return would still
over-read mihi's scratch buffer in `strlen` before reaching
`iam_copy_value`), but it does mean a NUL byte that survived past
`strlen` (i.e., embedded NUL in the cstring body) would render as
`?` — one less consequence to worry about.
