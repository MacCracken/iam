## Section A — mihi 1.0.0 surface delta vs v0.9.0 audit baseline

### CHANGELOG walk (mihi 0.7.0 → 1.0.0)

mihi shipped three release cuts between iam's v0.9.0 pin and the
v1.0.0 repin. The audit-relevant claims from
`https://raw.githubusercontent.com/MacCracken/mihi/1.0.0/CHANGELOG.md`:

- **0.7.0** — CI hardening only (distlib determinism gate, bench
  build gate, required-files list expansion). No source changes.
- **0.8.0** — acknowledgment cut for iam-0.9.0 RC integration. No
  source changes; `dist/mihi.cyr` regenerated only for the
  `# Version:` header stamp ("One-line diff; no module content
  changed").
- **1.0.0** — **API freeze**, no source changes ("module content
  byte-identical to 0.8.0 (which was itself byte-identical to
  0.7.0)"). One-line bundle header stamp.

**Implication for iam:** the probe semantics, cstring contracts,
buffer-fill behavior, and error-return shapes audited at v0.9.0 are
identical at the byte level in the mihi 1.0.0 bundle iam now links
against.

### Dep-tree movement

| -------------------- | --------------- | -------------- | ---------------- |
| `[deps.mihi].tag`    | `0.7.0`         | `1.0.0`        | major bump (audit-triggering) |

No transitive bump. mihi 1.0.0's own `cyrius.cyml` still pins

with the new mihi.

### External CVE / 0-day research (post v0.9.0 baseline)

Per the `feedback-security-audit-web-research` convention carried
from mihi's P(-1) — every audit pass must couple internal review
with external CVE research, not just one.

- **mihi-side**: mihi's own 0.6.0 audit landed three defensive
  parser hardenings (C-1, M-1, C-2: descending-range coercion and
  digit-overflow caps in `mihi_parse_cpu_range`,
  `mihi_parse_meminfo_kb`, `mihi_parse_uptime_secs`). These are
  carried forward into the 1.0.0 bundle iam now links — they make
  the parsers iam relies on *strictly safer* against adversarial
  `/proc` content. **Net effect on iam: reduced attack surface.**
- **ai-hwaccel-side**: pin unchanged at 2.2.6; no movement to
  audit.
- **Kernel-side**: the two AMD GPU kernel CVEs mihi 0.6.0
  documented as environmental
  ([CVE-2025-40289](https://nvd.nist.gov/vuln/detail/CVE-2025-40289),
  [CVE-2025-40288](https://nvd.nist.gov/vuln/detail/CVE-2025-40288))
  remain upstream-only — not mihi's to fix, not iam's to fix. They
  reach iam only through `mihi_gpu_*` on affected kernels.
  archaemenid's `7.0.5-arch1-1` is unaffected per mihi 0.6.0's note.
- **Cyrius toolchain**: pin unchanged at 6.0.1; no audit movement.

No new attack-surface-relevant CVE class introduced between v0.9.0
and v1.0.0.

## Section C — F-001 / F-002 carry-forward status

### F-001 — TTY-escape sanitization (RESOLVED, status unchanged)

`iam_copy_value` in `src/display.cyr:74` continues to backstop the
value column for every mihi-sourced byte. mihi 1.0.0 introduced no
new probe and no new cstring-typed return; the renderer fan-in
audited at v0.9.0 is unchanged. The 15 byte-exact assertions under
`test_group("display.cyr — F-001 TTY-escape sanitization")` pass
unchanged against the new mihi bundle.

### F-002 — `strlen` cstring trust (INFO, carries forward)

mihi 1.0.0's CHANGELOG explicitly froze the cstring contract for
v1.0 ("any signature / return-shape / error-semantics change is a
real `Breaking` and requires a major-version bump"). The
NUL-termination invariant iam relies on inside `iam_copy_value` and
`iam_render` is therefore now formally contract-bound on the mihi
side — F-002's residual risk is unchanged in magnitude but is now
versioned.

Recommendation: keep F-002 as INFO. Future mihi cuts that touch
cstring semantics would be a major bump and would re-trigger this
audit; no proactive iam-side change indicated.
