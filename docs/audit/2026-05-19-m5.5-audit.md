### Name collisions worth recording

So a future auditor running these queries doesn't have to re-prove
the disambiguation:

- **cyrius (the toolchain)** ≠ **Cyrus IMAP** / **Cyrus SASL** —
  homonym only. The Cyrus stack has a long CVE history (CVE-2022-24407
  among others); none of those advisories touch the cyrius compiler
  or stdlib.
- **ai-hwaccel (PCI detect)** ≠ **NVIDIA Container Toolkit** —
  recent NVIDIA-toolkit advisories (CVE-2024-0132, CVE-2025-23266) are
  container-escape bugs in OCI hook code, not in passive PCI
  enumeration libraries.

## Findings — status carried forward from M5

### F-001 — TTY escape sequences in mihi-returned strings (OPEN, LOW)

**Status carried**: still open. No mitigation landed between v0.5.0
and v0.6.0 (source unchanged). Remains the v1.0 freeze gate.

**Plan**: control-character / `\x1b` filter at the renderer boundary
in `src/display.cyr` — single pass over the value bytes inside
`iam_render` / `iam_render_buf` / `iam_render_kernel`, replacing any
byte `< 0x20` (except `\n`, which the renderer doesn't take from
mihi anyway) with `?`. Cost: sub-µs at realistic value lengths.
Lands either in v0.7.0 if scheduled with this audit's ship, or
ahead of the v0.9.0 RC at the latest.

### F-002 — `strlen` on mihi cstrings is trust-dependent (INFO)

**Status carried**: still informational. The dep pin on mihi has
not moved (0.7.0 throughout v0.5.0 → v0.6.0 → v0.7.0), so the
cstring invariant the renderer relies on is unchanged. No
iam-side action required.
